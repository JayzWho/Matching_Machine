# 前言：这个项目是什么？你会学到什么？

> 建议在开始写任何代码之前，先把这篇文档完整读一遍。
> 它不会教你技术细节，只负责让你建立一张"地图"，知道自己在做什么、为什么做、会收获什么。

---

## 一、用大白话说：我们在做什么？

你知道股票交易所是怎么工作的吗？

当你在券商 App 上下一个"以 100 元买入 100 股中石油"的订单，这个订单会被发送到交易所的服务器。交易所的服务器里有一个程序，它的工作就是：**把所有人的买单和卖单撮合在一起，找到买家和卖家都能接受的价格，成交**。

这个程序叫做 **撮合引擎（Matching Engine）**。

它需要处理的问题是：

- 每秒钟可能涌入几十万条订单
- 每一条都要在极短时间内处理完（纳秒到微秒级别）
- 必须保证"先到先得"，价格更优先、时间更早的订单优先成交
- 任何一条数据出错都可能引发严重的金融事故

**我们这个项目，就是用 C++ 手写一个这样的撮合引擎的核心部分。**

不连接真实交易所，不涉及真实资金，但模拟真实的逻辑和性能压力。

## 二、这个项目为什么有求职价值？

量化私募公司（做量化交易的基金）的技术岗位，每天面对的核心问题就是：**如何让系统更快、延迟更低、处理能力更强**。

你可以把他们的业务简化成这样理解：他们写程序，让程序比别人更快地发现市场上的机会，并在别人反应过来之前下单。快 1 毫秒，可能就多赚几百万。

所以这类公司非常在乎：

1. 你懂不懂"低延迟"编程（怎么让代码快）
2. 你懂不懂"并发"（多个线程同时工作不出错）
3. 你有没有实际测量过延迟，而不是靠感觉说"我觉得挺快的"

**这个项目的最终产出，是一组真实的延迟测量数据**，比如："我的撮合引擎处理一条订单的 P99 延迟是 XXX 纳秒，吞吐量是每秒 XXX 万笔"。

这种有数字的项目，在简历上远比"实现了一个 XXX 系统"有说服力。

## 三、做完这个项目，你会学到什么？

### 3.1 C++ 工程能力

| 技能 | 你学到什么 | 为什么值钱 |
|---|---|---|
| CMake 构建系统 | 如何管理多文件 C++ 项目和第三方依赖 | 量化公司全部用 CMake 或 Bazel |
| 内存管理 | 手写内存池，避免运行时 `malloc` | `malloc` 在高频场景下延迟不可控 |
| Lock-free 编程 | 用 `std::atomic` 写无锁数据结构 | mutex 加锁太慢，HFT 几乎不用 |
| 性能分析 | 用 `perf` 和火焰图找代码热点 | 不会分析就不知道优化哪里 |
| 延迟测量 | 用 RDTSC 精确测量纳秒级延迟 | 面试核心考点 |

### 3.2 计算机体系结构知识

你会在实践中真正理解，之前可能只是"听说过"的概念：

- **CPU Cache（缓存）**：为什么有些写法比另一些快 10 倍
- **False Sharing（伪共享）**：多线程代码慢的一个常见隐藏原因
- **内存屏障（Memory Barrier）**：多核 CPU 看到的内存顺序为什么不一样
- **分支预测**：CPU 猜测 if/else 走哪条路，猜错了要付出代价

### 3.3 金融基础知识

你会理解量化公司的实际业务背景：

- **订单簿（Order Book）**：交易所维护的买单/卖单列表
- **撮合规则（Price-Time Priority）**：价格优先，同价格下时间优先
- **做市商（Market Maker）**：一种量化策略，通过不断报买价和卖价赚差价
- **Tick 数据**：每一笔成交/挂单的原始数据流

## 四、开始之前，你需要自己去了解的关键词

下面这些词，你在开始动手之前，建议每个都搜一搜，看 5 分钟，形成基本印象就行，不需要深入。后面遇到了再深入学。

### 计算机方向

| 关键词 | 建议了解的程度 | 一句话解释 |
|---|---|---|
| **std::atomic** | 知道它是什么、为什么需要它 | C++ 标准库的原子操作类，用于无锁并发 |
| **Cache Line（缓存行）** | 知道它的大小（64字节）和作用 | CPU 从内存读数据的最小单位 |
| **False Sharing（伪共享）** | 知道它会导致什么问题 | 两个变量在同一缓存行上，多核修改时互相干扰 |
| **Memory Order（内存序）** | 知道有这个概念，了解 `memory_order_relaxed` 和 `memory_order_seq_cst` 的区别 | 控制编译器/CPU 指令重排的规则 |
| **Ring Buffer（环形缓冲区）** | 理解它的数据结构 | 一个首尾相连的固定大小数组，用于高效的生产者-消费者通信 |
| **SPSC** | 知道全称和场景 | Single Producer Single Consumer，单生产者单消费者，是最简单的无锁结构 |
| **内存池（Memory Pool）** | 知道它解决什么问题 | 提前分配一块内存，避免运行时频繁调用 `malloc`/`free` |
| **Google Benchmark** | 知道它是做什么的 | C++ 微基准测试框架，用来精确测量函数的执行时间 |
| **Google Test (GTest)** | 知道它是做什么的 | C++ 单元测试框架 |
| **perf** | 知道它是 Linux 性能分析工具 | 用来分析程序在哪里花了最多时间 |
| **CMake** | 知道它是 C++ 的构建系统 | 用来描述"怎么编译这个项目" |
| **RDTSC** | 知道它是 CPU 时间戳指令 | 读取 CPU 的时钟周期计数器，精度达纳秒级 |

### 金融方向

| 关键词 | 建议了解的程度 | 一句话解释 |
|---|---|---|
| **Order Book（订单簿）** | 理解它的结构 | 交易所维护的所有未成交买卖订单的列表，按价格排序 |
| **Bid / Ask** | 知道区别 | Bid 是买方愿意出的最高价，Ask 是卖方愿意接受的最低价 |
| **Spread（价差）** | 知道它是什么 | Ask 价 - Bid 价，做市商赚的就是这个差 |
| **市价单 / 限价单（Market Order / Limit Order）** | 理解区别 | 市价单：以当前最优价立刻成交；限价单：指定价格，等待对手方出现 |
| **撮合规则（Price-Time Priority）** | 理解规则 | 价格更优的先成交；同价格则先挂单的先成交 |
| **Tick 数据** | 知道是什么 | 每一次报价或成交的原始记录，是量化回测的原材料 |
| **量化策略** | 形成感性认识即可 | 用数学模型和程序代替人工做交易决策 |
| **HFT（高频交易）** | 知道核心特征 | 每秒下单成千上万次，靠速度优势获利，对延迟要求极苛刻 |
| **做市商（Market Maker）** | 知道它是什么 | 同时挂买单和卖单，赚取买卖价差，为市场提供流动性 |
| **回测（Backtesting）** | 知道概念 | 用历史数据验证交易策略的盈亏表现 |
| **延迟（Latency）** | 理解 P50/P99 的含义 | P99 延迟：100 笔里有 99 笔的处理时间低于这个数字 |

## 五、项目整体路线图（白话版）

```
第 1-3 周：打地基
  → 搭好 CMake 工程框架
  → 写好 Order（订单）数据结构
  → 实现 OrderBook（订单簿）—— 先用 mutex 保证正确性
  → 写好单元测试，保证逻辑没 bug

第 4-5 周：提速
  → 实现无锁 SPSC Ring Buffer（生产者-消费者通信用）
  → 实现内存池（替换 malloc）
  → 接入 RDTSC 延迟记录器

第 5-6 周：测量 & 优化
  → 用 Google Benchmark 建立延迟基线数字
  → 用 perf + 火焰图找出热点
  → 做针对性优化（缓存对齐、消除 false sharing）

第 7-8 周：组装完整流水线
  → FeedSimulator（模拟行情数据）→ SPSC Ring Buffer → OrderBook → TradeLogger
  → 端到端压力测试，拿到完整的延迟和吞吐量数字

第 9-10 周：包装输出
  → 写好 README（架构图 + benchmark 数据截图）
  → 准备简历 bullet points
  → 开始向量化私募投递
```
